SCENARIO Công ty thương mại điện tử tổng hợp — Minh Thành Co.
Quy mô: 25 nhân viên, doanh thu ~3 tỷ/tháng. Kinh doanh thiết bị văn phòng (máy tính, máy in, phụ kiện).

Kênh bán hàng: (1) Bán lẻ trực tiếp tại showroom, (2) Bán buôn cho các công ty qua nhân viên sale, (3) Đơn hàng online qua website (tích hợp sau).

Đặc thù: Khách hàng doanh nghiệp thường mua nhiều lần, có hạn mức công nợ riêng (credit limit). Một số sản phẩm có serial number (laptop, máy in). Có chính sách bảo hành 12–24 tháng. Giá bán theo 3 bảng giá: lẻ, buôn cấp 1, buôn cấp 2. Chiết khấu theo số lượng.

Pain point hiện tại: Đang quản lý bằng Excel + Zalo. Hay xảy ra: bán hàng không biết còn tồn kho không, quên không xuất hóa đơn, không biết khách nào đang nợ quá hạn, nhân viên không biết đơn hàng đang ở bước nào.
Mục tiêu workshop: Sau 4 giờ, nhóm phải có đủ tài liệu để một kỹ sư mới join có thể bắt đầu code ngay mà không cần hỏi thêm — đó là mức độ hoàn chỉnh cần đạt được.

    Tất cả trạng thái có thể có (Draft, Confirmed, Partially Delivered, Done, Cancelled) và điều kiện để chuyển giữa các trạng thái. Ai được phép trigger mỗi chuyển trạng thái.

Code Review · Phân tích
Code Review từ góc độ nghiệp vụ
Nhận một codebase ERP mô phỏng có chủ đích nhiều lỗi nghiệp vụ. Nhiệm vụ: tìm ra tất cả chỗ sai, giải thích tại sao sai và sửa đúng như thế nào.
3 giờ Cá nhân Thảo luận nhóm cuối

Dưới đây là code và schema của một ERP mini đang chạy thực tế. Mỗi đoạn code đều có ít nhất 2–3 lỗi nghiệp vụ. Kỹ sư cần tìm ra tất cả và ghi lại:

SNIPPET 1 API: Xác nhận đơn hàng 
// POST /api/orders/:id/confirm
async function confirmOrder(orderId) {
  const order = await db.orders.findById(orderId);
  
  // Kiểm tra tồn kho
  for (const item of order.items) {
    const product = await db.products.findById(item.productId);
    if (product.stock < item.quantity) {
      throw new Error('Không đủ hàng');
    }
    // Trừ tồn kho
    await db.products.update(item.productId, {
      stock: product.stock - item.quantity
    });
  }
  
  // Cập nhật trạng thái
  await db.orders.update(orderId, { 
    status: 'confirmed' 
  });
  
  return { success: true };
}
Gợi ý phân tích Snippet 1: Tìm ít nhất 5 lỗi nghiệp vụ. Gợi ý: nghĩ về atomicity, audit trail, credit limit, bút toán kế toán, và điều gì xảy ra nếu server crash giữa chừng vòng lặp for.
SNIPPET 2 Database Schema: Products & Orders 
CREATE TABLE products (
  id        INT PRIMARY KEY AUTO_INCREMENT,
  name      VARCHAR(255),
  price     DECIMAL(10,2),
  stock     INT DEFAULT 0
);

CREATE TABLE orders (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  customer    VARCHAR(255),
  total       DECIMAL(10,2),
  status      VARCHAR(20),
  created_at  DATETIME DEFAULT NOW()
);

CREATE TABLE order_items (
  id         INT PRIMARY KEY AUTO_INCREMENT,
  order_id   INT,
  product    VARCHAR(255),
  qty        INT,
  price      DECIMAL(10,2)
);
Gợi ý phân tích Snippet 2: Tìm ít nhất 8 vấn đề về schema. Gợi ý: foreign keys, soft delete, audit fields, pricing history, currency, tax, trường nào thiếu để làm báo cáo.
SNIPPET 3 Inventory: Xuất kho và tính giá vốn 
async function processDelivery(orderId) {
  const order = await db.orders.findById(orderId);
  
  for (const item of order.items) {
    // Xuất kho
    await db.products.decrement(item.productId, item.qty);
    
    // Ghi nhận giá vốn = giá mua mới nhất
    const lastPurchase = await db.purchases
      .findLast({ productId: item.productId });
    
    await db.cogs.insert({
      orderId,
      productId: item.productId,
      qty: item.qty,
      cost: lastPurchase.price  // FIFO? AVG? Không rõ
    });
  }
  
  // Gửi email cho khách
  await emailService.send(order.customerEmail, 
    'Đơn hàng đang giao');
    
  await db.orders.update(orderId, { status: 'delivered' });
}
Gợi ý phân tích Snippet 3: Tìm ít nhất 4 lỗi. Gợi ý: FIFO vs AVG không nhất quán, email blocking DB transaction, không có journal entry kế toán, không kiểm tra order status trước khi xử lý.

Tích hợp liên module & Event Design
Khi Sales Order được xác nhận, có bao nhiêu module bị ảnh hưởng? Thiết kế luồng dữ liệu liên module và chọn kiến trúc phù hợp cho SMB ERP.
4 giờ Nhóm 2–3 người Debate kiến trúc
Khi SO được xác nhận — ai cần biết?

Một hành động "xác nhận đơn hàng" kéo theo ít nhất 6 module cần cập nhật đồng thời hoặc tuần tự. Kỹ sư phải thiết kế rõ: module nào cập nhật ngay (synchronous), module nào có thể đợi (asynchronous), và khi có lỗi thì rollback như thế nào.

Cascade effect khi SO confirmed
⚡ Phải xảy ra ngay (Sync)
  • → Kho: Reserve số lượng hàng
  • → Kế toán: Ghi nhận doanh thu
  • → Công nợ: Tăng AR của khách
  • → Credit limit: Trừ hạn mức tín dụng
🔄 Có thể async (Queue)
  • → Kho: Tạo Delivery Order
  • → Email/SMS: Thông báo khách
  • → MRP: Kiểm tra cần sản xuất thêm
  • → Báo cáo: Cập nhật dashboard
Trade-off: Tight Coupling vs Event-Driven cho SMB ERP
Tight Coupling (Monolith)
Phù hợp cho SMB nhỏ (<5 devs)
Ưu điểm:
✓ Đơn giản, dễ debug
✓ Transaction ACID toàn bộ
✓ Không cần message broker
✓ Latency thấp
Nhược điểm:
✗ Khó scale từng module
✗ Một module lỗi → cả hệ thống
✗ Khó deploy riêng từng phần
Event-Driven (Microservices)
Phù hợp khi >10 devs, scale cao
Ưu điểm:
✓ Scale từng service riêng
✓ Fault isolation tốt
✓ Team độc lập
Nhược điểm:
✗ Phức tạp hơn nhiều
✗ Eventual consistency khó xử lý
✗ Cần message broker (Kafka, RabbitMQ)
Khuyến nghị cho SMB ERP: Bắt đầu với modular monolith — một codebase nhưng tổ chức theo module rõ ràng, giao tiếp qua internal events (không qua network). Khi cần scale module cụ thể mới tách ra. Đừng over-engineer ngay từ đầu cho khách hàng 20 người dùng.

    Resilience · Saga

Phần A — Lý thuyết nghiệp vụ (15 câu · 45 phút)
Kiến thức nền tảng ERP
Câu 1 / 30
Một công ty sản xuất nước uống vừa mở thêm chuỗi cửa hàng bán lẻ trực tiếp. Liệt kê tất cả module ERP cần thiết và giải thích tại sao mỗi module là cần thiết với loại hình kinh doanh kết hợp này.
Gợi ý: nghĩ đến cả hai luồng — sản xuất và bán lẻ — và sự giao thoa giữa chúng.
Các module cần: MRP/Sản xuất (BOM, Work Order, tính COGM cho nước uống), Kho NVL + Kho thành phẩm (hai kho riêng biệt), Mua hàng (nguyên liệu, bao bì), Bán hàng/POS (bán lẻ tại cửa hàng), Phân phối (chuyển thành phẩm từ nhà máy ra cửa hàng — internal transfer), Kế toán (ghi nhận COGM, doanh thu bán lẻ), Nhân sự (công nhân nhà máy + nhân viên cửa hàng khác nhau về cách tính lương), Báo cáo (P&L tách theo nhà máy vs chuỗi bán lẻ).
Câu 2 / 30
Giải thích 3-way matching. Tại sao đây là kiểm soát nội bộ quan trọng nhất trong module mua hàng? Kể một ví dụ gian lận cụ thể mà 3-way matching sẽ ngăn chặn được.
3-way matching đối chiếu PO (đặt hàng) + GRN (nhận hàng) + Vendor Invoice (hóa đơn NCC) trước khi cho phép thanh toán. Ví dụ gian lận: nhân viên mua hàng thông đồng với NCC, NCC xuất hóa đơn 110 cái nhưng chỉ giao 100 cái. Nếu không có 3-way matching, kế toán thanh toán theo hóa đơn 110 cái → thiệt hại 10 cái. Với 3-way matching: GRN ghi 100 cái → hệ thống chặn thanh toán → phát sinh cảnh báo chênh lệch.
Câu 3 / 30
Phương pháp tính giá tồn kho FIFO và AVG ảnh hưởng đến database schema như thế nào? Viết cấu trúc bảng cho mỗi phương pháp.
FIFO: Cần bảng stock_lots(id, product_id, lot_date, quantity_remaining, unit_cost). Khi xuất kho: lấy lô cũ nhất còn hàng trước, giảm quantity_remaining. AVG: Chỉ cần thêm avg_cost DECIMAL vào bảng products. Khi nhập hàng mới: avg_cost = (tổng giá trị hiện tại + giá trị nhập mới) / (tổng số lượng mới). Khi xuất: giá vốn = avg_cost tại thời điểm xuất.
Câu 4 / 30
Tại sao không nên xóa một Sales Order đã được xác nhận dù khách hàng yêu cầu hủy? Mô tả đúng quy trình hủy đơn hàng và các tác động đến dữ liệu liên quan.
SO đã confirmed đã: (1) reserve tồn kho, (2) có thể đã tạo DO, (3) đã ghi nhận trong hệ thống kế toán. Xóa SO sẽ gây inconsistency. Quy trình đúng: (1) Tạo Cancellation Request có lý do, (2) Phê duyệt nếu cần, (3) Giải phóng inventory reservation, (4) Nếu đã xuất hóa đơn: tạo Credit Note, (5) Nếu đã xuất kho: tạo Return Order, (6) SO chuyển trạng thái Cancelled — không xóa, lưu mãi để audit trail.
Câu 5 / 30
Thiết kế state machine hoàn chỉnh cho Purchase Order. Liệt kê tất cả trạng thái, điều kiện chuyển trạng thái, và ai có quyền trigger mỗi chuyển trạng thái.
Trạng thái: Draft (NV tạo, sửa tự do) → Submitted (NV submit, chờ duyệt) → Approved (Quản lý duyệt, gửi NCC) → Sent to Vendor (NCC nhận) → Partially Received (đã nhận một phần) → Fully Received (nhận đủ) → Closed (đã match invoice & thanh toán). Nhánh: từ Approved/Sent → Cancelled (GĐ duyệt hủy, có lý do). Từ Draft → Rejected (Quản lý từ chối). Ai trigger: NV (Draft→Submitted), Quản lý (Submitted→Approved/Rejected), Thủ kho (→Partially/Fully Received), KT (→Closed).

Phần B — Thiết kế hệ thống (10 câu · 45 phút)
Áp dụng vào thiết kế thực tế
Câu 16 / 30
Một khách hàng yêu cầu: "Tôi muốn xem báo cáo doanh thu theo từng nhân viên bán hàng, theo tháng, so sánh với cùng kỳ năm ngoái, breakdown theo nhóm sản phẩm." Thiết kế query và schema để đáp ứng yêu cầu này hiệu quả.
Cần: (1) sale_orders có salesperson_id, confirmed_date; (2) sale_order_lines có product_id, subtotal; (3) products có category_id; (4) dim_date có month, year để filter nhanh. Query: GROUP BY salesperson_id, MONTH(confirmed_date), YEAR(confirmed_date), category_id. Với "cùng kỳ năm ngoái": UNION hoặc subquery với YEAR = current_year - 1. Tối ưu: tạo materialized view hoặc fact table cập nhật daily nếu data lớn.
Câu 17 / 30
Thiết kế hệ thống phân quyền (RBAC) cho ERP có 5 vai trò: Admin, Giám đốc, Kế toán, Nhân viên bán hàng, Thủ kho. Phân quyền phải hỗ trợ: record-level (chỉ thấy data của bộ phận mình), state-based (không sửa được document đã approved), và field-level (không thấy giá vốn nếu không phải kế toán).
Schema: roles(id, name), permissions(id, role_id, resource, action, conditions JSONB). Conditions ví dụ: {"department": "own", "status": ["draft","submitted"]}. Implement: (1) Record-level: thêm department_id vào mọi document, filter ở tầng repository; (2) State-based: check status trong permission conditions trước khi cho phép action; (3) Field-level: GraphQL field resolvers hoặc serializer loại bỏ field nhạy cảm theo role. Lưu permission check vào audit_log.
Câu 20 / 30
Khách hàng phàn nàn rằng khi nhân viên bán hàng tạo đơn hàng giờ cao điểm, hệ thống rất chậm. Nguyên nhân có thể là gì từ góc độ nghiệp vụ ERP và bạn sẽ giải quyết như thế nào?
Nguyên nhân nghiệp vụ: (1) Khi tạo SO đang check stock và lock rows trong inventory table → blocking khác; (2) Đang tính price realtime từ nhiều bảng phức tạp (pricelist, discount, tax); (3) Sync journal entry creation blocking; (4) Missing indexes trên date + status columns. Giải pháp: (1) Separate inventory reservation vào async queue — show "reserved" optimistically; (2) Cache pricelist vào memory, invalidate khi admin sửa; (3) Journal entry async sau khi SO confirmed; (4) Add composite index (customer_id, status, created_at).

Phần C — Phân tích tình huống (5 câu · 30 phút)
Case study và quyết định thực tế
Câu 26 / 30
Bạn đang code review và thấy đồng nghiệp viết: await db.orders.delete(orderId) sau khi khách hàng yêu cầu hủy đơn. Đơn hàng này đã ở trạng thái "confirmed" và đã tạo Delivery Order. Bạn sẽ comment gì và đề xuất code đúng như thế nào?
Comment: "❌ Hard delete nghiêm trọng — vi phạm audit trail. SO đã confirmed đã tạo DO và có thể đã ghi journal entry. Xóa sẽ gây orphan records và mất lịch sử kiểm toán." Cách đúng: (1) Validate: order.status === 'confirmed'; (2) Validate: DO chưa được pick (nếu đã pick thì cần return flow khác); (3) Begin transaction; (4) Cancel DO liên quan; (5) Release inventory reservation; (6) Reverse journal entries nếu có; (7) Update order.status = 'cancelled', order.cancelled_at = now(), order.cancelled_by = userId, order.cancel_reason = reason; (8) Commit. Không bao giờ delete.
Câu 30 / 30
Một khách hàng SMB mới yêu cầu ERP có tất cả tính năng: Sản xuất, Kho, Mua hàng, Bán hàng, Kế toán, Nhân sự, CRM, E-commerce, Logistics. Timeline: 3 tháng. Team của bạn có 4 người. Bạn sẽ tư vấn như thế nào?
Không thể làm tốt tất cả trong 3 tháng với 4 người. Tư vấn: (1) Phân loại must-have vs nice-to-have: hỏi khách hàng "Nếu chỉ có 3 module, bạn chọn gì?" — thường là Kho + Bán hàng + Kế toán cơ bản; (2) Phased delivery: MVP 3 tháng (3 module core), Phase 2 sau 3 tháng tiếp (Mua hàng + Nhân sự), Phase 3 (Sản xuất + CRM); (3) Tránh over-engineering: không build sản xuất đầy đủ nếu khách hàng chỉ có 2 sản phẩm; (4) Integrate với tool sẵn có: dùng MISA cho kế toán, tích hợp API thay vì tự build; (5) Ký hợp đồng theo phase với acceptance criteria rõ ràng.
